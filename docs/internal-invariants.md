# 내부 불변식 주석 작성 규칙

SkidMC의 내부 계약은 함수 작성자가 직접 접근하는 상태와 직접 호출하는 함수의 문서만 확인해도 구현을 판단할 수 있게 한다. 계약은 구현 설명이 아니라 시그니처와 코드만으로 알 수 없는 필수 정보만 보충한다.

## 지역적 추론

함수를 구현할 때 확인하는 범위는 다음으로 제한한다.

- 직접 읽거나 수정하는 상태의 `INVARIANT:`, `THREADING:`
- 현재 함수와 직접 호출하는 함수의 `REQUIRES:`, `ENSURES:`, `FAILURE:`

피호출 함수의 내부 상태, 하위 호출, 같은 상태를 변경하는 다른 함수는 확인하지 않는다. 모든 함수는 직접 접근한 상태의 불변식을 보존하고 실패 후 호출자 판단에 필요한 결과를 자신의 계약으로 노출한다.

상태를 수정하는 함수는 반환, 다른 함수 호출, 이벤트·콜백 호출과 예외 전파 전에 불변식을 다시 성립시킨다. 원자적 전이 중 일시적으로 불변식이 깨질 수 있지만 그 상태를 다른 코드에 노출하지 않는다.

## 계약 블록

| 선언 대상 | 허용 블록 | 의미 |
| --- | --- | --- |
| 상태 | `INVARIANT:` | 상태가 관찰 가능한 모든 시점에 유지되는 조건 |
| 상태 | `THREADING:` | 상태 접근이 허용되는 스레드와 동시성 조건 |
| 함수 | `REQUIRES:` | 시그니처로 표현되지 않는 호출 전제 |
| 함수 | `ENSURES:` | 정상 반환 후의 비자명한 결과와 부수 효과 |
| 함수 | `FAILURE:` | 실패 후 남는 비자명한 상태와 호출자 의무 |

```bash
rg 'INVARIANT:|THREADING:|REQUIRES:|ENSURES:|FAILURE:'
```

별도의 `RECOVERY:`는 사용하지 않는다. 내부에서 복구하고 정상 반환하면 결과를 `ENSURES:`에, 호출자에게 실패를 전달하며 비자명한 상태가 남으면 `FAILURE:`에 작성한다.

## 최소 작성 기준

다음 순서로 필요한 계약인지 판단한다.

1. 시그니처, 타입, nullability, 함수명 또는 기본 언어 동작으로 알 수 있으면 쓰지 않는다.
2. 안전한 상태 수정이나 함수 호출에 필요하지 않으면 쓰지 않는다.
3. 생략했을 때 간접 상태나 하위 구현을 열어봐야 하는 정보만 쓴다.

내부 구현 과정, 자명한 반환값, 일반적인 예외 전파와 프로젝트 구조상 항상 보장되는 환경은 반복하지 않는다. 반대로 계약을 더 줄였을 때 직접 의존성 아래의 구현을 확인해야 한다면 필요한 정보까지 제거한 것이다.

## 상태 선언부

상태 선언부에는 `INVARIANT:`와 `THREADING:`만 쓴다. 한 필드가 상태를 소유하면 그 필드에, 여러 필드가 하나의 상태를 구성하면 가장 가까운 공통 소유 타입에 쓴다. 상태를 소유하지 않는 handler 타입에는 계약 블록을 쓰지 않는다.

```kotlin
/**
 * INVARIANT:
 * - 저장된 관계는 MOUNT_EARLY가 완료된 뒤 추가됐다.
 * - 하나의 rider ID는 최대 하나의 saddle ID에 속한다.
 *
 * THREADING:
 * - 렌더 스레드에서만 접근하고 변경한다.
 */
private val handledPassengerIdsByKartId =
    mutableMapOf<Int, MutableSet<Int>>()
```

타입이나 컬렉션 구조가 보장하는 내용, 상태가 만들어진 과정과 구현 전략은 쓰지 않는다. 여러 조건이 같은 제약에서 파생되면 가장 근본적인 조건만 남긴다. 조건은 긍정형이며 검사 가능하게 작성한다.

## 함수 선언부

함수 선언부에는 필요한 블록만 선택해 쓴다. 상태 불변식과 상태의 `THREADING:`은 함수에 복사하지 않는다.

### REQUIRES

파라미터 타입과 nullability로 표현할 수 없는 조건만 쓴다. 엔티티 사이의 관계, 수명 주기 단계와 정확한 Mixin 주입 시점 등이 해당한다. 구조적으로 항상 보장되는 호출 스레드는 반복하지 않는다.

### ENSURES

호출자가 관찰할 수 있는 비자명한 내용만 쓴다.

- 공유 상태나 전달받은 객체의 변경
- 이벤트 발행 여부와 순서
- 기존 참조의 무효화
- Boolean, nullable 결과 등이 구분하는 상태

내부 캐시나 구현용 컬렉션의 변경 과정은 쓰지 않는다.

### FAILURE

일반적인 예외 전파는 쓰지 않는다. 다음처럼 실패 후 결과가 호출자 판단에 영향을 줄 때만 쓴다.

- 일부 부수 효과가 남는다.
- 예외와 관계없이 별도 정리나 무효화를 수행한다.
- 재시도 가능 여부가 달라진다.
- 호출자가 추가 처리를 해야 한다.

불변식이 깨진 상태로 성공을 반환하거나 실패를 숨기지 않는다. 복구할 수 없으면 부분 상태를 폐기하거나 명시적인 무효 상태로 전환한 뒤 실패를 전달한다.

## 상태 계약 연결

직접 접근하는 상태와의 관계가 명확하지 않으면 `@see`로 상태 선언부를 연결할 수 있다. 이는 구현자와 리뷰어를 위한 탐색 링크이며 함수 계약은 아니다.

- 직접 접근하는 상태만 연결한다.
- 피호출 함수의 내부 상태는 연결하지 않는다.
- 관계가 자명한 함수에는 기계적으로 추가하지 않는다.
- Mixin 주입점 같은 일반적인 KDoc 링크는 별도로 사용할 수 있다.

## 구현 절차

### 구현 전

1. 현재 함수에 계약 블록이 있으면 확인한다.
2. 직접 접근할 상태의 계약을 확인한다.
3. 직접 호출할 함수에 계약 블록이 있으면 확인한다.
4. 피호출 함수의 실패 후 상태를 처리하거나 현재 함수 계약에 노출해야 하는지 결정한다.

그보다 하위의 구현이나 전체 호출 그래프는 추적하지 않는다. 기존 주석과 구현이 다르면 어느 쪽이 올바른 계약인지 먼저 결정한다.

### 구현 중

- 직접 읽는 상태는 선언된 불변식을 전제로 사용한다.
- 직접 수정한 상태는 제어권을 넘기기 전에 불변식을 복구한다.
- 피호출 함수는 선언된 계약만 만족시켜 호출한다.
- 이벤트와 콜백에 불변식이 깨진 중간 상태를 노출하지 않는다.
- 예외 경로의 정리가 필요하면 `finally`나 안전한 변경 순서를 사용한다.
- Mixin handler에서는 한 번만 호출되는 private 함수로 주입 흐름을 분산하지 않고, 여러 흐름이 공유하는 상태 연산만 함수로 분리한다.

### 구현 후

1. 현재 함수의 계약과 직접 수정한 상태의 불변식을 만족하는지 확인한다.
2. 피호출 함수의 전제를 만족하고 실패 후 상태를 처리했는지 확인한다.
3. 새로 발견한 지속적 규칙은 상태 선언부에 추가한다.
4. 시그니처, 이름 또는 기본 동작과 중복되는 주석을 제거한다.
5. 변경한 전이와 대표적인 제거·월드 변경·복구 경로를 테스트한다.

상태 표현이나 불변식 자체를 바꾸는 경우에는 해당 상태의 직접 참조를 검색한다. 중요한 가변 상태를 `private`으로 유지하고 가변 컬렉션을 외부에 노출하지 않아 검토 범위를 상태 소유자 안으로 제한한다. 기존 코드는 관련 기능을 수정할 때 점진적으로 이 규칙을 적용한다.

## 복구와 이벤트

- 외부 입력 불일치를 안전하게 정리할 수 있으면 관련 ID, 생략한 동작과 정리 결과를 `WARN`으로 남긴다.
- 정상적인 pending 상태에는 경고를 남기지 않고 이후 입력에서 재시도한다.
- 이벤트 계약에 필요한 객체가 없으면 이벤트는 생략하되 확인된 내부 관계는 정리한다.
- 이벤트 중 조회되어야 하는 상태는 콜백이 끝날 때까지 유효하게 유지한다.
- 콜백 예외와 관계없이 정리해야 하는 상태가 있으면 `finally`를 사용하고 필요한 실패 후 상태만 `FAILURE:`에 쓴다.
